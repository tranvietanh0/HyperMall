## Key Insights
- Highest risk is edge/control-plane drift: discovery locator, missing route policy binding, hardcoded registry creds, permissive shared config.
- Build/config drift is real: `backend/ai-service/pom.xml` exists but parent omits module; config-server coverage misses seven services plus AI.
- Frontend auth is fragmented: `ProtectedRoute` unused, route ownership split, refresh flow races, storage writes leak across UI.
- Best path is progressive: secure ingress first, then align source-of-truth config/build, then consolidate frontend auth/routing, then add release guards.

## Requirements
- Produce phased implementation path that minimizes blast radius and keeps service contracts stable.
- Prioritize security/control-plane fixes before config/build drift, before frontend consolidation, before ops hardening.
- Keep changes YAGNI/KISS/DRY: prefer explicit ownership over new infra or broad rewrites.
- Define validation gates for gateway auth, config coverage, build inclusion, frontend guard behavior, and startup health.

## Architecture
- Gateway becomes single enforced edge: explicit routes, auth/rate-limit policy attached per route group, discovery locator disabled or tightly scoped.
- Config Server becomes non-secret source of truth for shared/service config; env or secret store owns credentials and signing material.
- Parent build and config inventory become canonical service roster; every runnable service must exist in both or be explicitly retired.
- Frontend route tree becomes layout-first with one auth module handling storage, refresh lock/queue, logout cleanup, and role guards.

## Success Criteria
- No unauthenticated service path reaches protected downstream APIs via autogenerated gateway routes.
- Parent build, config-server files, and documented service inventory agree.
- Frontend uses one guarded route tree and one refresh/logout flow.
- Shared config defaults are safe for non-dev environments and observable by smoke checks.

## Risk Assessment
- Main risk is auth regressions at gateway and frontend after policy consolidation.
- Secondary risk is surfacing hidden dependencies when config/build drift is corrected.
- Mitigation: phase by boundary, keep temporary compatibility shims, validate each phase before moving on.

## Security Considerations
- Remove hardcoded registry creds from code; do not replace with new git-tracked secrets.
- Enforce explicit gateway auth and route policy before relying on downstream service checks.
- Review shared config defaults for schema mutation, verbose logs, and actuator exposure.

## Next steps
- Start with Phase 1 and keep later phases blocked until gateway auth and control-plane ownership are verified.
